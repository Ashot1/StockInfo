

# Agentic LLM Development Workflow (Cursor + Codex + GitHub Projects)

## Что мы строим
Собираем **агентную систему разработки**, где одну большую задачу можно задать как **Epic**, после чего система помогает:

- автоматически **декомпозировать** Epic на подзадачи
- вести работу в **Kanban** (прозрачные статусы)
- выполнять задачи **параллельно несколькими агентами**
- создавать PR и проводить **обязательное ревью**
- накапливать **долгоживущую память проекта** через базу знаний

Фокус: pet‑проект, скорость и автономность. Без требований к безопасности.

---

## Выбранный стек

### IDE / управление агентами
- **Cursor** — основной командный пункт (Agent mode) + MCP‑клиент.
- **Codex** — исполнитель задач (параллельные сессии, PR, тесты).

### Трекер задач / Kanban
- **GitHub Issues** — единицы работы.
- **GitHub Projects** — Kanban‑доска (Status‑колонки) и обзор прогресса.

### Интеграции для агентов
- **GitHub MCP Server** — чтобы агенты могли читать/создавать issues, управлять PR и синхронизировать статусы.

### База знаний (Knowledge Base)
- **AnythingLLM** — индексирует документацию/решения/Context Packs и служит “памятью проекта”.

## Ключевые идеи системы

### 1) Kanban — источник правды по прогрессу
Прогресс отслеживается **не в голове и не в чатах**, а в GitHub Projects.

### 2) Context Pack на каждую задачу (обязательно)
У каждой Issue есть ссылка на **один markdown‑документ в репозитории** — *Context Pack*.

Содержимое Context Pack (минимум):
- Goal / Definition of Done (DoD)
- Архитектура / решение (коротко)
- Файлы / точки входа
- Тест‑план
- Риски / долги

Этот документ автоматически индексируется в AnythingLLM, чтобы:
- агенты могли подхватывать контекст без твоего участия
- решения не терялись между сессиями
- ревьюер мог проверять PR по DoD и тест‑плану

### 3) Автоматизация “жизненного цикла” задач
Базовый конвейер:
- PR открыт → карточка / Issue переходит в **In Review**
- PR смёрджен → карточка / Issue переходит в **Done**

Реализуется GitHub Actions + (при необходимости) правила/агент.

### 4) Reviewer‑агент как обязательная стадия
Отдельная роль агента: **не пишет код**, только ревьюит PR по правилам проекта.

## Артефакты в репозитории (документы системы)
В папке `AI-workflow/` ведём набор документов:

1. `Overview.md` — этот файл (что строим, стек, принципы).
2. `Management-Process.md` — как вести Kanban, как описывать задачи, что прикладывать.
3. `Development-Process.md` — как разрабатывать: ветки/PR/тесты/ревью/мердж.
4. `Roles.md` — роли (Planner/Workers/Reviewer), обязанности, ожидания, входы/выходы.
5. `Setup-Guide.md` — пошаговый гайд по созданию системы с нуля.

В корне репозитория:
- `AGENTS.md` — правила и чеклисты для всех агентов (DoD, стиль, тесты, PR‑процесс, обновление статусов и Context Packs).

## Целевой workflow (высокий уровень)
1. Ты формулируешь Epic.
2. Planner‑агент:
   - уточняет требования
   - создаёт набор Issues (8–15) + проставляет зависимости/параллельность
   - создаёт/заполняет Context Packs
3. Worker‑агенты параллельно:
   - берут Issues
   - реализуют
   - открывают PR
   - обновляют Context Packs
4. Reviewer‑агент:
   - ревьюит PR по `AGENTS.md` и DoD
   - либо просит правки, либо approve
5. Автоматизация двигает статусы, AnythingLLM индексирует знания.
